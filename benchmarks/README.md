# benchmarks — inference speed on real phones

This directory holds benchmark reports generated by `scripts/bench.sh`. Each
report is a single markdown file per device containing device metadata,
model info, and per-prompt timings. The file name is a slug derived from
`ro.product.manufacturer` + `ro.product.model` + `ro.board.platform`.

```
benchmarks/
├── README.md                             — this file
├── prompts.json                          — fixed prompt set (versioned)
├── lge-lm-g850-sm8250.md                 — LG V60 ThinQ, Snapdragon 865
└── <your-device>.md                      — submit via PR!
```

## The ask

Every phone is different. Snapdragon vs Dimensity vs Exynos vs Tensor, 4 GiB
vs 12 GiB RAM, 2019-era vs 2024-era silicon — they all run Ollama at
dramatically different speeds. A claim like "qwen2.5:1.5b runs fine on an
old phone" is only useful with **real numbers on real hardware**.

If you've run `install-ollama.sh` on your phone and have a working Ollama
server, running the benchmark and submitting the markdown file is **one
command + one PR**:

```bash
# In Termux, with the server already running in another session
bash ~/ollama-pocket/scripts/bench.sh

# This writes benchmarks/<your-device-slug>.md. Open a PR with that file.
```

The prompt set is fixed at `prompts.json` (version 1) so numbers are
comparable across devices without caveats. If you change the prompts, bump
the version — old benchmarks become stale the moment the fixed set changes.

## Contribution workflow

1. Start the Ollama server and pull the model you want to benchmark:
   ```bash
   bash ~/ollama-pocket/scripts/start-ollama.sh --wifi
   # in another Termux session:
   proot-distro login debian -- ollama pull qwen2.5:1.5b
   ```
2. Run the bench (multiple runs recommended for warm median stability):
   ```bash
   bash ~/ollama-pocket/scripts/bench.sh --runs 3
   ```
3. The script writes `benchmarks/<device-slug>.md`. Inspect it — the
   Warm median should be > 0 tok/s and the Per-prompt detail table should
   have one row per prompt × run.
4. Commit just that one file and open a PR using the
   `benchmark_contribution` issue template.
5. The reviewer may ask you to add notes about your testing conditions
   (plugged in vs battery, thermal state, background apps, etc.) — those
   matter a lot for reproducibility.

## Flags

`bench.sh` supports:

- `--model <name>`  — bench a different model (default: `qwen2.5:1.5b`)
- `--host <url>`    — override Ollama host (default: `http://localhost:11434`)
- `--runs N`        — repeat each prompt N times (default: 1). Use 3+ for
  stable warm-median numbers.
- `--output <path>` — override the report path
- `--dry-run`       — print config, don't call Ollama (useful for checking
  device detection)

## What's measured

Each prompt call returns Ollama's timing fields in the response:

- `eval_count` — number of output tokens produced
- `eval_duration` — nanoseconds spent on the output
- `load_duration` — nanoseconds spent loading the model (first call only)
- `prompt_eval_count` / `prompt_eval_duration` — input processing (not
  included in tok/s since it's usually tiny)

**tok/s** is computed as `eval_count / (eval_duration / 1e9)` — the
throughput of *output token generation only*, excluding prompt evaluation
and model load. This is the number you'd quote as "how fast does this phone
generate text".

The **cold** run is the first prompt after `bench.sh` starts (it unloads any
in-memory model via `keep_alive: 0` before running, so the first call is
always a cold load). Subsequent runs are **warm** — the model stays in
memory, no load cost.

## What's NOT measured (yet)

- **Battery impact** — you'd need an external power meter for meaningful
  numbers. Anecdotal: running Ollama at full tilt drains ~20% battery per
  hour on a 4000 mAh phone.
- **Thermal throttling** — bench.sh runs 5 prompts × N runs in ~1 minute,
  which is too short for most phones to thermally throttle. For sustained
  throughput, run `--runs 20` or higher.
- **Peak RAM** — bench.sh takes a single VmRSS snapshot at the end of the
  run. It's not a peak — a background sampler would be needed for that.
- **Quantization-aware comparisons** — every benchmark reports which
  quantization the model uses (from `/api/tags`), but we don't normalise
  across quants. Q4_K_M and Q8_0 on the same model will have different tok/s.

These are all reasonable follow-ups. Opening PRs for them is welcome.
